// Capabilities

Key Features

🤖
Multi-Phase AI Agent
Recon → Enum → Vulnerability Mapping → Exploitation Planning — fully automated pipeline with phase-aware LLM orchestration.
🔒
Three-Layer Scope Guard
CIDR enforcement at LLM planning, backend API, and weapon gateway — defence-in-depth prevents out-of-scope execution.
📋
Tamper-Evident Audit
SHA-256 hash-chained event log — every action cryptographically linked. Detect tampering by re-computing the chain.
🧠
Local LLM — 100% Offline
qwen2.5:14b for deep reasoning + llama3.2:3b for fast tasks via Ollama. No API keys. No external calls. Zero data egress.
📚
RAG Knowledge Base
Vector search with nomic-embed-text embeddings. Ingest custom playbooks and pentest methodologies for context-aware planning.
🔐
mTLS Inter-Node
Mutual TLS between Command Node and Weapon Node. Certificate-pinned. JWT-scoped tool gateway.
✅
Approval Workflows
High-risk tools (nuclei, gobuster, os_detection) require explicit operator approval before execution.
⚡
Real-Time Streaming
Live tool output streamed via WebSocket ExecutionBus + NDJSON. Sub-50ms latency from weapon node to UI.
🏠
Fully Self-Hosted
Supabase GoTrue auth on-premises. All user data, findings, and artifacts stay on your own servers.
// Installation

Step-by-Step Setup Guide

Complete deployment from bare Proxmox VMs to a running PentAI Pro system.

01

Provision Proxmox VMs

Create three VMs on your Proxmox host in the 172.20.32.0/18 VLAN. Command Node needs 25 GB RAM and 400 GB storage.

# Expand LVM after resizing disk in Proxmox
sudo growpart /dev/sda 3
sudo pvresize /dev/sda3
sudo lvextend -l +100%FREE /dev/ubuntu-vg/ubuntu-lv
sudo resize2fs /dev/mapper/ubuntu--vg-ubuntu--lv
df -h /   # confirm ~392 GB
02

Install Docker on Command Node

Docker and Docker Compose are required. Install using the official convenience script.

curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER && newgrp docker
03

Clone & Generate mTLS Certificates

Clone the repository and generate the CA + client/server certificate chain for inter-node communication.

git clone https://github.com/ANIS151993/pentai-pro.git
cd pentai-pro
chmod +x scripts/generate-certs.sh
./scripts/generate-certs.sh
04

Deploy Supabase Self-Hosted Auth

Run all 13 Supabase containers locally. No Supabase Cloud account needed. Email autoconfirm is enabled for lab use.

git clone https://github.com/supabase/supabase.git ~/supabase-docker
cd ~/supabase-docker/docker
cp .env.example .env
# Edit .env: set JWT_SECRET, KONG_HTTP_PORT=8543, ENABLE_EMAIL_AUTOCONFIRM=true
docker compose up -d
# Verify: curl http://172.20.32.74:8543/auth/v1/health
05

Configure Environment & Build

Set all secrets and build the backend + frontend Docker images.

cp .env.example .env
# Edit .env: PENTAI_AUTH_JWT_SECRET, PENTAI_SUPABASE_JWT_SECRET, PENTAI_CORS_ALLOW_ORIGINS

# Configure frontend
cat > frontend/.env.local << EOF
NEXT_PUBLIC_SUPABASE_URL=http://172.20.32.74:8543
NEXT_PUBLIC_SUPABASE_ANON_KEY=<your-anon-key>
NEXT_PUBLIC_API_BASE_URL=http://172.20.32.74:8000
EOF

docker compose build backend frontend
docker compose up -d
06

Pull AI Models via Ollama

Three models are required. The 14b model is the main reasoning engine (~9 GB download).

docker exec pentai-pro-ollama-1 ollama pull qwen2.5:14b-instruct-q4_K_M
docker exec pentai-pro-ollama-1 ollama pull llama3.2:3b-instruct-q4_K_M
docker exec pentai-pro-ollama-1 ollama pull nomic-embed-text
docker exec pentai-pro-ollama-1 ollama list
07

Deploy Weapon Node (Kali Linux)

Install pentest tools and deploy the mTLS Tool Gateway on the Weapon Node.

# On Kali (172.20.32.68)
sudo apt install -y nmap nuclei gobuster whatweb sslscan
go install -v github.com/projectdiscovery/httpx/cmd/httpx@latest
go install -v github.com/projectdiscovery/dnsx/cmd/dnsx@latest

# From Command Node
./scripts/setup-weapon-node.sh
08

Verify & Use the System

All services are up. Navigate to the PentAI Pro UI and create your first engagement.

# Health check
curl http://172.20.32.74/api/v1/healthz

# Access points:
# PentAI Pro    → http://172.20.32.74
# Supabase Studio → http://172.20.32.74:3001
# API Docs      → http://172.20.32.74/docs
// User Manual

How to Use PentAI Pro

A complete walkthrough — from first login to reading your final penetration test report.

01
Access

Login to the System

Open your browser and navigate to the PentAI Pro dashboard. Use the account created during setup, or register a new operator account.

  • Go to https://apts.marcbd.site (or your custom domain)
  • Click Sign Up to create a new operator account
  • Enter your email and a strong password — no email confirmation needed (auto-confirm is on)
  • After login, you land on the Operator Workspace dashboard
TIP: The Core_Health panel on the right shows the live status of all system components. All four rows should show a green check mark within a few seconds.
02
Engagement

Create a New Engagement

An engagement defines the scope and authorisation for a penetration test. You must create one before running any tools.

  • Click the blue + New_Task button (top right) or the orange ⚡ FAB button (bottom right)
  • Fill in the Engagement Name (e.g. Lab Network Q2 2025)
  • Enter the Target Scope — one or more CIDR ranges (e.g. 172.20.32.59/32)
  • Set the Operator Name and optional Notes
  • Click Create — the engagement status is set to Active
WARN: The system will reject any tool invocation that targets an IP outside the declared scope. Always set scope accurately.
03
Dashboard

Understand the Dashboard

The Operator Workspace gives you a real-time operational picture at a glance.

  • KPI Grid — shows active operations, critical findings, total hosts discovered, and security posture index
  • Operational Queue — lists all engagements with their current agent run status
  • Core_Health panel — monitors Control Plane API, PostgreSQL, Weapon Node connection, and Ollama LLM service
  • Live Intel feed — shows the four most recent findings sorted by time
  • Risk Distribution chart — bar chart of findings by severity across all engagements
TIP: The dashboard auto-polls for health status every 30 seconds. You do not need to refresh the page manually.
04
Reconnaissance

Run Automated Reconnaissance

Open an engagement and start the AI agent. The agent decides which tools to run and in what order.

Engagements List → Select Engagement → Start Agent
  • From the Operational Queue, click an engagement row to open it
  • Click Start Agent Run — the agent enters the Reconnaissance phase
  • Watch the live terminal panel for real-time tool output (nmap, httpx, whatweb)
  • The agent automatically advances through: Recon → Enumeration → Vulnerability Mapping
  • Each tool invocation is scope-checked at three layers before execution
05
Exploitation

Approve or Deny Exploit Actions

PentAI Pro never auto-exploits without your approval. When the agent reaches the exploitation phase, it pauses and waits.

  • A Pending Approvals badge appears on the engagement row in the queue
  • Open the engagement — you will see the proposed exploit action with full details
  • Review the target IP, CVE ID, risk level, and the exact command that will be run
  • Click Approve to let the agent proceed, or Deny to skip that action
  • The agent logs your decision and continues to the next task
WARN: Approved exploits run against real targets on the Weapon Node. Only approve actions targeting systems you are authorised to test.
06
Findings

Review Findings & Evidence

All discovered vulnerabilities are logged in the Findings panel with tamper-evident audit records.

  • Navigate to the engagement → Findings tab
  • Each finding shows: severity badge, CVE reference, affected host, description, and evidence snippet
  • Click a finding row to expand the full technical detail and raw tool output
  • Findings are colour-coded: Critical / High / Medium / Low / Info
  • Every finding is linked to its audit log entry — you can verify the SHA-256 chain at any time
07
Inventory

Explore Host Inventory

Every host and open service discovered during the engagement is stored in the Inventory.

  • Navigate to the engagement → Inventory tab
  • The Hosts table shows IP, hostname, OS guess, and discovery time
  • The Services table shows port, protocol, banner, and version string
  • Click a host row to see all services detected on that host
  • The KPI Grid on the main dashboard shows the total host count across all engagements
TIP: Inventory data is built incrementally — each agent run adds new hosts and services without overwriting previous discoveries.
08
Report

Generate the PDF Pentest Report

When the engagement is complete, generate a professional PDF report with one click.

  • Open the engagement → scroll to the bottom → click Generate Report
  • The agent uses the reasoning LLM to write narrative findings and remediation advice
  • Report includes: executive summary, methodology, findings table, evidence screenshots, CVSS scores
  • Click Download PDF to save the report locally
  • Reports are stored in /app/artifacts/ on the Command Node container

// Complete Operator Workflow

Login → Create Engagement + Set Scope → Start Agent Run → Recon & Enumeration (auto) → Vuln Mapping (auto) → Approve Exploits (manual) → Post-Exploitation (auto) → Review Findings → Download Report

All tool invocations are scope-validated at three layers and recorded in a tamper-evident SHA-256 hash chain. No raw shell commands are ever executed — every action is structured JSON dispatched through the Tool Gateway on the Weapon Node.

// Common Questions & Troubleshooting

Q: Core_Health shows OFFLINE after login?

Wait 5–10 seconds and the panel will auto-refresh. The health poller runs every 30s. If it stays red, check that all Docker containers are running: docker compose ps

Q: Agent run completes but no findings?

Verify the Target Node is reachable from the Weapon Node. SSH to Kali and run ping 172.20.32.59. Also confirm the scope CIDR matches the target IP exactly.

Q: Ollama LLM shows UNKNOWN?

Check that the Ollama container is healthy: docker compose ps ollama. Then verify models are pulled: docker exec pentai-pro-ollama-1 ollama list

Q: How do I add a new target for a second engagement?

Create a new engagement with the new target's CIDR. Each engagement is isolated — scope enforcement is per-engagement. You can run multiple engagements simultaneously.

Q: How do I access the Supabase admin dashboard?

Navigate to http://172.20.32.74:3001 on your local network. Default credentials are in ~/supabase-docker/.env under DASHBOARD_USERNAME and DASHBOARD_PASSWORD.

Q: Where are the API docs?

FastAPI auto-generates interactive docs at https://apts.marcbd.site/api/docs (Swagger UI) and /api/redoc. You can test all endpoints directly from the browser.

// Safety Design

Security Architecture

Non-negotiable security invariants embedded at every layer of the system.

Layer 1 — LLM Planning
Scope-Bounded Proposal
Every target the AI proposes is validated against engagement scope_cidrs before any tool invocation is generated. Out-of-scope proposals are rejected at the planner level.
Layer 2 — Backend API
CIDR + Approval Gate
Backend re-validates CIDR membership on every tool invocation request. High-risk operations (nuclei, gobuster) require an explicit operator approval before a JWT is issued to the Weapon Node.
Layer 3 — Weapon Gateway
mTLS + Token Verification
Weapon Node re-validates scope on arrival over mutual TLS. JWT signature, audience, and scope are verified before any subprocess is launched. Defence-in-depth against compromised command layer.
Audit Chain
SHA-256 Hash Chaining
Every security event (tool validation, execution, approval, finding) is linked: hash[n] = SHA256(event + prev_hash). Genesis hash = "0" * 64. Tamper detection by re-computing the full chain.
Prompt Injection Defense
Output Sandboxing
All tool output passed to the LLM is wrapped in <tool_output trust="untrusted"> tags. The LLM is instructed to treat this content as data only — preventing adversarial targets from hijacking the agent.

Abstract
Automated penetration testing has long remained a distant goal for the security community, hampered by the difficulty of encoding expert reasoning into deterministic rule systems. The emergence of large language models (LLMs) capable of contextual reasoning opens a new path. This paper introduces PentAI Pro, a self-hosted, multi-phase automated penetration testing framework that orchestrates LLM-driven agents across physically isolated virtual machines. The system implements a structured attack lifecycle — Reconnaissance, Enumeration, Vulnerability Mapping, and Exploitation Planning — guided by a retrieval-augmented generation (RAG) knowledge base and enforced by a three-layer scope validation mechanism. All components run on-premises using the 14-billion-parameter qwen2.5:14b-instruct model via Ollama. Evaluation demonstrates an average reconnaissance planning latency of 18.4 seconds, full-phase pipeline completion under 4 minutes, and 100% out-of-scope rejection across all tested scenarios.
Keywords — penetration testing automation, large language models, retrieval-augmented generation, security orchestration, mutual TLS, audit logging, scope enforcement
I. Introduction
Penetration testing remains one of the most labour-intensive disciplines in information security. A skilled practitioner must plan a reconnaissance strategy, enumerate live services, identify vulnerabilities, prioritise exploitation paths, and document findings — all while operating strictly within an agreed scope of engagement...
Prior efforts at automation have taken two broad approaches. Script-based frameworks such as Metasploit and OpenVAS codify known attack patterns into deterministic modules, but they lack the adaptive reasoning required to chain observations into coherent attack narratives...
II. Related Work
A. Automated Vulnerability Scanning: Tools such as Nessus, OpenVAS, and Nuclei automate individual scanning tasks using signature-based detection. While effective at finding known vulnerabilities, they do not reason about attack chains or produce narrative findings...
B. LLMs for Security Tasks: Happe and Cito demonstrated that GPT-4 can engage in multi-turn penetration testing dialogues. PentAI Pro extends this by deploying LLM reasoning entirely on-premises using open-weight models, eliminating data egress concerns...
III. System Architecture
PentAI Pro is deployed across three isolated virtual machines on a Proxmox hypervisor sharing an internal VLAN (172.20.32.0/18). The Command Node (172.20.32.74) runs all orchestration, inference, and data storage; the Weapon Node (172.20.32.68) runs the tool execution gateway; the Target Node is the assessed system.
